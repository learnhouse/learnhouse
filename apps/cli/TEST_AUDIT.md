# LearnHouse CLI — Test Genuineness Audit

Generated by the `cli-test-audit` multi-agent workflow (mutation testing + adversarial
static review + feature-gap analysis), then **empirically re-verified by hand** against the
real worktree before acting. All findings below were confirmed by actually breaking the code
and watching the suite.

## Method

- **Mutation testing** — 30 deliberate code-breakages across every unit; each applied in an
  isolated copy with the full in-process suite run. A *real* test must fail when the behavior
  it covers is broken; a mutation that "escapes" (suite still green) proves a fake or missing test.
- **Adversarial static review** — 20 agents read each unit's tests hunting for over-mocking,
  assertion-free "it resolves" tests, tautologies, and flaky/real-network tests.

## Headline (corrected)

- **30 mutations: 24 caught, 6 escaped (20% escape rate).** The suite is mostly genuine — the
  harness *did* catch 24 deliberate breakages — but had **6 proven blind spots** plus **1 flaky
  real-network test** and a cluster of **assertion-light backup/restore tests**.
- (The workflow's auto-generated summary mis-stated this as "6/6 = 100% escape" because the
  synthesis agent was only handed the 6 escapes, not the 24 catches, and separately "verified"
  against the wrong branch (`dev`, which lacks these tests). Both errors are corrected here.)

## Proven gaps (mutation escapes) — all 6 now FIXED

Each row: I broke the code, confirmed the old suite stayed green (escape), added a real
assertion, then confirmed the suite now fails on the same breakage (caught).

| # | unit | behavior that wasn't really tested | the new assertion that catches it |
|---|------|------------------------------------|-----------------------------------|
| 1 | `version-check.ts` `compareVersions` | comparison only needed the **major** segment to pass; minor/patch never exercised | `checkForUpdates` test with a **patch-bumped** version must print "Update available" |
| 2 | `docker.ts` `dockerComposeUpRetry` | the **default `attempts = 3`** was never exercised (tests always passed it explicitly) | call with no `attempts`; fail twice then succeed; assert it took 3 attempts |
| 3 | `network.ts` `getPublicIp` | the **anchored** `^…$` IP regex — the only negative case had no IP substring at all | body `"<p>1.2.3.4</p>"` (embeds an IP) must still return `null` |
| 4 | `backup.ts` `createBackup` | the **`--clean --if-exists`** pg_dump flags (idempotent restore) weren't asserted at unit level | `expect(dockerExecToFile).toHaveBeenCalledWith('learnhouse-db-dep1', 'pg_dump -U learnhouse --clean --if-exists learnhouse', …)` |
| 5 | `env-check.ts` `checkDevEnv` | the **`COLLAB_PORT=4000`** dev default was a test *input fixture*, never an output assertion | assert the generated `apps/collab/.env` actually contains `COLLAB_PORT=4000` |
| 6 | `docker.ts` `autoDetectDeploymentId` | the **`name=learnhouse-app-`** filter (mocks ignore the arg, so a wrong filter passed silently) | assert the `docker ps` command string contains `name=learnhouse-app-` |

## Assertion-light tests — strengthened

The static pass flagged a cluster of backup/restore tests that proved a function *ran* but not
that it did the *right* thing. Fixed the highest-value ones:

- **`--restore extracts and restores`** — was `resolves.toBeUndefined()` only. Now asserts psql
  actually ran: `dockerExecFromFile('learnhouse-db-dep1', 'psql -U learnhouse -d learnhouse', …)`.
- **`--restore cancels when declined`** — was "some `ProcessExit` thrown". Now asserts the exit
  code is **0** (clean cancel) **and** that `dockerExecFromFile` was never called.
- **`--restore refuses external DB`** / **`…db not running`** — now assert exit code **1** and that
  no restore was attempted (the guard fired *before* touching the database).
- **`interactive menu can route to restore`** — previously had **zero `expect()`**. Now asserts the
  menu routed to restore (psql ran).

## Flaky test — fixed

- **`checkTcpConnection — timeout`** dialed RFC-5737 `192.0.2.1` and was timing/network dependent
  (could spuriously "connect" on intercepting networks — it was the lone intermittent failure during
  this audit). Rewritten to mock `net.createConnection` with a socket that never resolves →
  deterministically exercises the timeout branch (resolve `false` + socket destroyed), plus a new
  positive case (connects → `true`).

## Remaining lower-priority edges (not yet addressed)

From the static gap analysis — candidates for a follow-up pass:

- `backup.ts` success output strings ("Backup complete!", size line, restore hint) are unasserted.
- `backup.ts` archive timestamp colon/dot sanitization in the filename is only indirectly covered.
- `backup.ts` does **not** remove `tmpDir` on a tar-step failure (a latent temp-dir leak — a real
  code smell, not just a test gap).
- `config.ts` `configCommand` `:443` port-suffix suppression branch (`useHttps:true && httpPort===443`)
  is not exercised.

## Net effect

The six proven blind spots are closed and re-verified (the same mutations are now caught), the flaky
test is deterministic, and the weakest backup/restore assertions now pin exit codes and exact command
arguments. The suite went from "green but partly hollow" to "green and load-bearing" on these paths.
